## 18.1 为什么需要分区实验？

### 18.1.1 问题背景

当实验涉及：
- **大模型**：内存占用大，无法一次加载所有数据
- **多任务**：多个数据集需要评测
- **多方法**：多个方法需要对比
- **多模型组合**：异构多智能体系统

如果一次性运行所有样本：
- GPU 显存不足 (OOM)
- 实验时间过长
- 中间失败导致全部重来

### 18.1.2 解决方案：分区实验

将实验**分区 (Partition)** 进行：

```
原始实验:
  所有数据 → 一个实验任务 → OOM

分区实验:
  数据分区 1 → 实验任务 1
  数据分区 2 → 实验任务 2
  数据分区 3 → 实验任务 3
  ...
  最后合并结果
```

## 18.2 launch_partition_run.sh 概览

### 18.2.1 脚本位置

**位置**：`scripts/launch_partition_run.sh`

### 18.2.2 功能

这是一个 Bash 启动脚本，用于：
1. 解析环境变量配置
2. 生成实验配置
3. 调用 `partition_runner.py` 管理执行

| 变量 | 说明 | 示例 |

| `RUN_NAME` | 实验名称 | `exp_vwormhole_qwen_lfm` |
| `GPUS` | GPU 列表 | `0,1,2,3,4,5,6,7` |
| `GPUS_PER_JOB` | 每个实验的 GPU 数 | `1` |
| `TASKS` | 任务列表 | `gsm8k,arc_easy,gpqa` |
| `METHODS` | 方法列表 | `vision_latent_mas_codec_new` |
| `AGENT_MODEL_NAMES` | 模型名称 | `"Qwen,Qwen3-VL-2B,LFM2.5-VL-1.6B"` |
| `VISION_CODEC_PATH` | 编解码器路径 | `checkpoints/merged.pt` |

## 18.4 分区管理机制

### 18.4.1 partition_runner.py 概览

**位置**：`scripts/partition_runner.py`

### 18.4.2 两种调度器

```python
# 支持两种调度模式
SCHEDULER="${SCHEDULER:-queue}"  # 或 "wave"

# wave: 严格按 (task, method) 分组
# queue: 队列式，每个 GPU 自主领取任务
```

### 18.4.3 文件命名规范

```
<jsonl_dir>/<task>_<method_tag>_world_<N>_partition_<k>.jsonl
```

示例：
```
preds_jsonl/gsm8k_vcodec_world_8_partition_0.jsonl
preds_jsonl/gsm8k_vcodec_world_8_partition_1.jsonl
...
```

## 18.5 GPU 资源分配

### 18.5.1 分配策略

```bash
GPUS=0,1,2,3,4,5,6,7    # 8 张 GPU
GPUS_PER_JOB=1             # 每任务使用 1 张 GPU
```

可以并行运行 **8 个任务**。

### 18.5.2 多 GPU 任务

```bash
GPUS_PER_JOB=2  # 每任务使用 2 张 GPU
```

适用于模型较大，单卡放不下的情况。

## 18.6 OOM 重试策略

### 18.6.1 问题

大模型推理时可能出现 **OOM (Out of Memory)**：

```
CUDA out of memory. Tried to allocate 12.00 GiB
```

### 18.6.2 解决思路

OOM 时，**减少 batch size** 并重试：

```bash
OOM_RETRY_LEVELS=12,8,4,2,1
```

这表示：
1. 首先尝试 batch_size=12
2. 如果 OOM → 减少到 8
3. 如果还 OOM → 减少到 4
4. 以此类推，直到 1

### 18.6.3 自动调整

```bash
AUTO_GENERATE_BS=1  # 启用自动 batch size 调整
```

## 18.11 常见问题与解决

### 18.11.1 GPU 资源不足

```
Error: GPU requested but not available
```

**解决**：减少 `GPUS_PER_JOB` 或选择空闲的 GPU。

### 18.11.2 所有分区都 OOM

```
Error: OOM even at minimum batch size
```

**解决**：
1. 减少 `LATENT_STEPS`
2. 使用更小的模型
3. 手动拆分更多分区

### 18.11.3 分区结果丢失

**解决**：
1. 检查 `LOG_DIR` 中的错误日志
2. 重新运行失败的分区
3. 使用 `SUMMARY_JSON` 检查进度

## 18.12 本课小结

### 关键要点

1. **分区实验**：将大任务拆分，避免 OOM，支持容错
2. **启动脚本**：`launch_partition_run.sh` 通过环境变量配置
3. **调度器**：queue（队列式）和 wave（严格分组）
4. **OOM 重试**：`OOM_RETRY_LEVELS=12,8,4,2,1` 自动降级
5. **结果合并**：`merge_partition_jsonl.py` 合并分区结果
