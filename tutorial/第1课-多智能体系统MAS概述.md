## 课程目标

- 理解什么是多智能体系统及其应用场景

- 认识传统文本通信的局限性

## 1.1 什么是多智能体系统 (Multi-Agent System)

**多智能体系统 (Multi-Agent System, MAS)** 是由多个自主或半自主的智能体 (Agent) 组成的计算系统，这些智能体能够相互通信、协作或竞争来完成复杂任务。

### 智能体的定义

在 LLM 语境下，**智能体 (Agent)** 是一个具有以下能力的实体：

- **感知 (Perception)**：接收输入（如用户问题、上下文）
- **推理 (Reasoning)**：利用内部模型进行思考和推理
- **行动 (Action)**：生成输出（如回答、计划、代码）
- **通信 (Communication)**：与其他智能体交换信息

### MAS 的核心特征

| 特征 | 描述 |
|------|------|
| **自主性** | 每个智能体独立运作，有自己的决策能力 |
| **局部视图** | 智能体只持有部分信息，没有全局视图 |
| **去中心化** | 没有中央控制器，智能体自主协作 |
| **社会性** | 智能体之间可以通信、协调、协商 |

---

## 1.2 LLM 驱动的多智能体系统

随着大语言模型 (LLM) 的发展，研究者开始将 LLM 作为智能体的"大脑"，构建 LLM 驱动的 MAS。

### 为什么使用 LLM？

传统 MAS 中的智能体通常是规则-based 或小型 ML 模型，能力有限。而 LLM 提供了：

- **强大的推理能力**：Chain-of-Thought、ReAct 等推理策略
- **丰富的知识**：预训练涵盖广泛的世界知识
- **灵活的交互**：自然语言理解和生成
- **涌现能力**：复杂任务上的零样本/少样本学习

### LLM-MAS 的典型架构

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   Planner    │ ───▶ │   Critic    │ ───▶ │   Refiner   │
│  (计划者)    │      │  (批评者)    │      │  (改进者)    │
└─────────────┘      └─────────────┘      └─────────────┘
                                                  │
                                                  ▼
                                           ┌─────────────┐
                                           │   Judger    │
                                           │  (评判者)    │
                                           └─────────────┘
```

#### 角色说明

| 角色 | 功能 | 典型输出 |
|------|------|----------|
| **Planner** | 制定解决计划 | "为了解决这个问题，我需要三步..." |
| **Critic** | 评估计划质量 | "这个计划有漏洞，因为..." |
| **Refiner** | 改进计划 | "基于反馈，我提出改进方案..." |
| **Judger** | 给出最终答案 | "综合分析，最终答案是..." |

## 1.3 传统文本通信的局限性

在传统 LLM-MAS 中，智能体之间通过**离散文本**进行通信。这种方式存在两大核心问题：

### 1.3.1 运行时开销 (Runtime Overhead)

文本通信需要经过完整的**生成-传输-解析**流程：

```
Agent A 生成文本 ──▶ 网络传输 ──▶ Agent B 解析文本
     │                                      │
     ▼                                      ▼
  [LLM Forward]                    [LLM Forward + Parsing]
     │                                      │
     └──────────────────────────────────────┘
                     总延迟
```

每个 token 的生成都需要消耗大量计算资源，而文本传输涉及完整的序列化和反序列化过程。

### 1.3.2 信息量化损失 (Information Quantization Loss)

文本本质上是**离散符号序列**，将连续的推理状态压缩为离散文本会造成信息损失：

```
连续推理状态 [0.234, -0.891, 0.567, ...]  ──▶  文本 "The answer is 42"
                                                ↑
                                          信息损失
```

这种损失在高带宽场景下尤为明显——模型产生的丰富推理轨迹被压缩成寥寥数语。

### 1.3.3 量化损失的数学理解

假设模型的隐藏状态为 $h \in \mathbb{R}^d$，文本 token 序列为 $t \in \{1, 2, ..., V\}^L$：

**编码过程**：
$$h \xrightarrow{\text{LLM}} t$$

**解码过程**：
$$t \xrightarrow{\text{LLM}} \hat{h}$$

由于 LLM 的输出是离散的，$h \approx \hat{h}$ 通常不成立，存在量化误差：

$$\epsilon = \| h - \hat{h} \|_2$$

## 1.4 现有方法的局限性

除了文本通信，研究者还提出了多种替代方案，但各有缺陷。

### 1.4.1 纯潜在空间转移 (Latent Space Transfer)

**思想**：直接传递连续潜在状态，而非离散文本

```
Agent A 潜在状态 ──▶ 传输 ──▶ Agent B 潜在状态
     h_A         ──▶    ──▶        h_B
```

**问题**：需要 Sender 和 Receiver 具有相同的架构（**同构**），否则潜在空间不兼容：

$$h_A \in \mathbb{R}^{d_A}, \quad h_B \in \mathbb{R}^{d_B}$$

不同维度的向量无法直接对应！

### 1.4.2 成对翻译器 (Pair-specific Translators)

**思想**：为每对模型训练专用翻译器

```
Model A ──▶ [A→B Translator] ──▶ Model B
Model A ──▶ [A→C Translator] ──▶ Model C
Model B ──▶ [B→C Translator] ──▶ Model C
...
```

**问题**：对于 $N$ 个模型，需要 $O(N^2)$ 个翻译器，扩展性极差！

### 1.4.3 现有方法对比

| 方法 | 优点 | 缺点 |
|------|------|------|
| 文本通信 | 实现简单、兼容性好 | 延迟高、信息损失大 |
| 潜在空间转移 | 高带宽、低延迟 | 只能同构通信 |
| 成对翻译器 | 支持异构 | $O(N^2)$ 复杂度 |

## 1.5 本课程的核心目标

**Vision Wormhole** 提出了一个根本性的解决方案：

> **将视觉接口作为"通用端口"，实现模型无关的、文本无关的跨智能体通信。**

### 核心洞察

视觉-语言模型 (VLM) 天然具有**两种输入通道**：

1. **文本通道**：接收文本 token
2. **视觉通道**：接收图像 token

关键发现：**视觉通道可以被劫持为高速通信端口！**

### 设计目标

| 目标 | 实现方式 |
|------|----------|
| **模型无关** | 所有模型共享同一个"视觉接口" |
| **文本无关** | 不需要序列化/反序列化文本 |
| **高效** | 连续潜在空间传输，无量化损失 |
| **可扩展** | $O(N)$ 复杂度，而非 $O(N^2)$ |

## 1.6 Hub-and-Spoke 拓扑

Vision Wormhole 采用 **Hub-and-Spoke（中心-辐射）** 拓扑结构：

```
                    ┌──────────────┐
                    │   Universal  │
                    │  Visual Codec │
                    │    (Hub)      │
                    └──────────────┘
                     /      |      \
                    /       |       \
               Encoder   Encoder   Encoder
                  │        |        │
              ┌───────┐┌───────┐┌───────┐
              │ Qwen  ││ Gemma ││ LFM   │
              │  -VL  ││  3-VL ││ 2.5-VL│
              └───────┘└───────┘└───────┘
```

### 工作原理

1. **编码 (Encode)**：每个模型的潜在状态通过自己的 `Encoder` 映射到通用空间
2. **传输 (Transfer)**：在通用空间中进行传输
3. **解码 (Decode)**：通过 `Decoder` 将通用表示转换回目标模型的视觉 token
4. **注入 (Inject)**：将视觉 token 直接注入接收者的视觉通道

### 复杂度分析

| 方案 | 翻译器数量 | 复杂度 |
|------|-----------|--------|
| 成对翻译器 | $N(N-1)/2$ | $O(N^2)$ |
| Hub-and-Spoke | $2N$ | $O(N)$ |

## 1.7 本课小结

### 关键要点

1. **多智能体系统 (MAS)** 由多个自主智能体组成，通过协作解决复杂问题
2. **LLM-MAS** 利用大语言模型作为智能体的推理引擎
3. **文本通信** 虽然简单，但存在**高延迟**和**信息损失**两大问题
4. **潜在空间转移** 可避免文本量化损失，但受限于同构架构
5. **Vision Wormhole** 提出用**视觉通道**作为通用通信端口，实现 $O(N)$ 的异构通信
